
# Agent Workflow

Головна точка входу для агента під час роботи над MVP Task Manager.

Документ описує:
- де знаходиться контекст проєкту;
- які файли читати перед роботою;
- як знаходити поточний sprint і task;
- які обмеження діють під час реалізації;
- коли і як формувати task-документацію, фінальне review та проєктну документацію.

---

## 1. Quick Start For Agent

Перед будь-якою реалізацією:

1. Відкрити цей файл.
2. Ознайомитися з project workspace.
3. Знайти поточний активний sprint у ex.: `.project_office/sprints/`.
4. Відкрити `plan.md` поточного sprint.
5. Знайти поточну активну task у `tasks/` поточного sprint.
6. Перевірити, чи є пов'язана документація в `implementation_docs/` поточного sprint.
7. Переглянути потрібні design або reference матеріали з `.project_office/design/`, якщо вони стосуються поточної task.
9. Коротко описати поточний стан sprint і task.
10. Підтвердити, який sprint і яка task активні.
11. Запропонувати короткий план реалізації.
12. Дочекатися підтвердження перед змінами в коді, якщо scope, поведінка або архітектурне рішення не є однозначними.

Не починати імплементацію одразу після читання task, якщо вимоги неповні або допускають кілька валідних підходів.

## 3. Context Sources

### Roadmap і Backlog

Особисті нотатки автора проєкту. Використовувати тільки як фоновий контекст. Не порівнювати з ними реалізацію, не виводити з них вимоги і не реалізовувати їх зміст без явного завдання.

### Sprints

Кожен sprint має власну папку. Використовувати `plan.md` як джерело:
- загального scope;
- очікуваного результату sprint;
- черговості tasks;
- відкритих питань;
- стану реалізації.

### Tasks

Task є основною одиницею реалізації всередині sprint.

Використовувати task як джерело:
- конкретного scope реалізації;
- acceptance criteria;
- локальних вимог;
- очікуваних змін у коді;
- статусу конкретної роботи.

Для кожної завершеної і прийнятої користувачем task потрібно створити або оновити окрему документацію в:

`.project_office/sprints/<sprint>/implementation_docs/`

Task-документація не замінює фінальну документацію в `docs/`. Вона є проміжним технічним контекстом для sprint.

### Design

`.project_office/design/` містить скріншоти, референси та дизайн-матеріали.

Використовувати як:
- джерело очікуваного UI;
- перевірку відповідності реалізації;
- уточнення поведінки окремих елементів інтерфейсу.

Не переносити дизайн-рішення в код, якщо вони не стосуються поточної task або не були погоджені.

## 5. Sprint Rules

Правила:
- кожен sprint знаходиться в окремій папці у `.project_office/sprints/`;
- `plan.md` є головним файлом плану sprint;
- `tasks/` містить task-файли sprint;
- `implementation_docs/` містить документацію завершених і прийнятих користувачем tasks;
- sprints додаються та уточнюються поступово під час роботи над проєктом;
- список sprints не є фінальним і може змінюватися;
- не відкривати та не виконувати всі sprints одразу;
- працювати тільки з поточним sprint і поточною task;
- не переходити до наступного sprint без підтвердження;
- не створювати нові sprints або службові файли без погодження;
- не позначати sprint завершеним без апруву користувача.

### Sprint Naming

Формат папок:

Активний sprint: `001 - Short Description`

Завершений sprint: `X - 001 - Short Description`

Sprint вважається завершеним тільки після явного підтвердження користувача. Після такого підтвердження папку sprint перейменувати, додавши префікс `X - `.

У корені папки sprint зберігати:
- `plan.md` як основний план;
- `tasks/` як список task-файлів sprint;
- `implementation_docs/` як документацію завершених і прийнятих користувачем tasks;
- `review.md` або інший погоджений review-файл тільки після запиту на фінальне review.

## 6. Task Rules

Задачі знаходяться в:

`.project_office/sprints/<sprint>/tasks/`

Правила:
- кожна task знаходиться в окремому файлі;
- tasks додаються та уточнюються поступово під час роботи над sprint;
- список tasks не є фінальним і може змінюватися;
- не відкривати та не виконувати всі tasks одразу;
- працювати тільки з поточною task, яку явно поставлено або погоджено;
- не переходити до наступної task без підтвердження;
- не створювати нові tasks або підзадачі без погодження;
- не позначати task виконаною без апруву користувача;
- створювати або оновлювати відповідний файл у `implementation_docs/` тільки після явного підтвердження користувача, що task виконана і правок більше немає.
- після явного підтвердження користувача, що task виконана і правок більше немає, перейменувати task-файл, додавши префікс `X - ` до назви.

### Task Naming

Формат файлів:

Активна task: `001 - Short Description.md`

Завершена task: `X - 001 - Short Description.md`

Task вважається завершеною тільки після явного підтвердження користувача, що task виконана і правок більше немає. Після такого підтвердження файл task перейменувати, додавши префікс `X - `.

Перед початком роботи над task:
- знайти активну task;
- прочитати тільки потрібний task файл;
- перевірити, чи існує пов'язана документація в `implementation_docs/`;
- працювати тільки в межах поточної task.

## 7. Implementation Workflow

Для кожної task всередині поточного sprint використовувати цей процес.

### Step 1. Receive Scope

Дії:
- отримати sprint і конкретну task;
- знайти відповідну папку у `.project_office/sprints/`;
- прочитати `plan.md`;
- прочитати поточний task файл у `tasks/`;
- перевірити пов'язану документацію в `implementation_docs/`;
- ознайомитися з потрібним roadmap, design та project context;
- визначити область змін.

Результат:
- агент розуміє, що саме треба зробити;
- агент розуміє, які файли або модулі можуть бути зачеплені;
- відкриті питання винесені явно, а не перетворені на припущення.

### Step 2. Planning

Перед початком реалізації:
- не додумувати вимоги;
- не приймати архітектурні рішення без підтвердження;
- підготувати короткий план реалізації;
- описати підхід;
- назвати можливі ризики або відкриті питання;
- отримати підтвердження перед змінами, якщо є неоднозначність.

Якщо вимоги неоднозначні, спочатку поставити питання.

### Step 3. Implementation

Після підтвердження або за наявності однозначного scope:
- реалізувати тільки погоджений scope;
- уникати змін поза межами поточної task;
- дотримуватись існуючих патернів проєкту;
- віддавати перевагу мінімальним, точковим змінам;
- не робити cleanup або refactoring, якщо це не потрібно для поточної task;
- не змінювати поведінку суміжних частин без явного погодження.

### Step 4. Verification

Після реалізації:
- запустити релевантні перевірки, якщо вони доступні;
- якщо перевірки не запускалися, явно сказати чому;
- перевірити, що зміни залишаються в межах погодженого scope;
- за потреби оновити статус або нотатки у task файлі або `plan.md`, але тільки якщо це було погоджено.

### Step 5. Review With User

Після завершення реалізації:
- коротко описати внесені зміни;
- показати результат або пояснити, де його перевірити;
- повідомити про запущені перевірки;
- запитати, чи потрібні правки.

Не створювати фінальний review-файл автоматично. Фінальне review робити тільки за запитом користувача.

### Step 6. Task Implementation Documentation

Після реалізації task не створювати і не оновлювати документацію task автоматично.

Створити або оновити документацію task у `implementation_docs/` тільки після того, як користувач явно підтвердив, що task виконана і правок більше немає.

Після такого підтвердження також перейменувати task-файл у `tasks/`, додавши префікс `X - ` до назви.

Шлях для документації task:

`.project_office/sprints/<sprint>/implementation_docs/`

Рекомендований формат файлу:

`001 - Task Name.md`

У документації task описати:
- що було реалізовано;
- які файли або модулі змінено;
- важливі рішення та компроміси;
- які перевірки були запущені;
- що варто знати наступному агенту.

Після створення документації згенерувати commit message через `/generate-commit-message` і показати результат у чаті.

Ця документація є обов'язковою для кожної завершеної і прийнятої користувачем task. Вона створюється в межах sprint і використовується як база для фінального review та проєктної документації.

### Step 7. Final Review On Request

Фінальне review формується тільки після прямого запиту користувача.

Review-файл зберігати в корені папки поточного sprint.

Рекомендований шлях:

`.project_office/sprints/<sprint>/review.md`

У review описати:
- що було реалізовано в межах sprint;
- які tasks завершені;
- які файли або модулі змінено;
- які перевірки були запущені;
- які ризики або обмеження залишилися;
- що потрібно врахувати перед переходом до наступного sprint.

Фінальне review формувати з урахуванням task-документації з `implementation_docs/`.

### Step 8. Project Documentation On Request

Проєктну документацію формувати тільки після того, як:
- sprint або його погоджена частина завершені;
- потрібні task-документи в `implementation_docs/` створені або оновлені;
- правок більше немає або користувач прийняв поточний результат;
- користувач явно попросив сформувати документацію.

Документацію зберігати в корені проєкту:

`docs/`

У документації описати:
- поточну поведінку або реалізований функціонал;
- важливі рішення та обмеження;
- основні файли або модулі;
- що варто знати наступному агенту або розробнику.

Фінальна документація проєкту формується на основі погоджених результатів sprints і task-документації, а не на основі неперевірених проміжних припущень.

## 8. Agent Constraints

Завжди:
- працювати в межах поточного sprint і поточної task;
- берегти існуючі патерни backend/frontend проєкту;
- питати перед неоднозначними або архітектурними рішеннями;
- коротко пояснювати план перед реалізацією;
- явно повідомляти, які перевірки були запущені;
- створювати або оновлювати `implementation_docs/` тільки після явного підтвердження користувача, що task виконана і правок більше немає;
- явно відділяти факти з файлів від припущень.

Не робити без погодження:
- перехід до наступного sprint;
- перехід до наступної task;
- створення нових sprints, tasks або підзадач;
- перейменування sprint-папок;
- перейменування task-файлів на `X - ...` до явного підтвердження користувача, що task виконана і правок більше немає;
- великі refactoring зміни;
- зміни поза межами поточної task;
- фінальне review;
- документацію в `docs/`;
- зміну структури `.project_office/`.
