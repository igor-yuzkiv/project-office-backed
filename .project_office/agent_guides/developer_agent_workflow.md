# Developer Agent Workflow

Головна точка входу для агента під час роботи над MVP Task Manager.

Структура workspace, шляхи та шаблони — у `.project_office/agent_guides/project_workspace.md`.

---

## 1. Quick Start

Перед будь-якою реалізацією:

1. Прочитати цей файл і `project_workspace.md`.
2. Знайти активний sprint у `.project_office/sprints/` і відкрити його `plan.md`.
3. Знайти активну task у `tasks/` цього sprint.
4. Перевірити пов'язану документацію в `implementation_docs/`.
5. Переглянути потрібні матеріали з `.project_office/design/`, якщо стосуються task.
6. Коротко описати стан sprint і task, підтвердити що активне.
7. Запропонувати план реалізації.
8. Дочекатися підтвердження перед змінами, якщо scope, поведінка або архітектура неоднозначні.

Не починати імплементацію, якщо вимоги неповні або допускають кілька валідних підходів.

---

## 2. Context Sources

**Roadmap і Backlog** — особисті нотатки автора. Тільки фоновий контекст. Не виводити з них вимоги.

**Sprint plan** (`plan.md`) — джерело загального scope, очікуваного результату, черговості tasks, відкритих питань.

**Task file** (`tasks/<task>.md`) — джерело конкретного scope, acceptance criteria, локальних вимог, очікуваних змін у коді.

**Implementation docs** (`implementation_docs/`) — проміжний технічний контекст для sprint. Не замінює фінальну документацію в `docs/`.

**Design** (`.project_office/design/`) — джерело очікуваного UI. Не переносити дизайн-рішення в код, якщо вони не стосуються поточної task або не погоджені.

---

## 3. Sprint And Task Rules

### Загальні правила

- Працювати тільки з поточним sprint і поточною task.
- Sprints і tasks додаються поступово; списки не фінальні.
- Не відкривати та не виконувати всі sprints/tasks одразу.
- Не переходити до наступного sprint або task без підтвердження.
- Не створювати нові sprints, tasks або підзадачі без погодження.
- Не позначати sprint або task завершеними без апруву користувача.

### Naming

| Type           | Active                       | Completed                        |
| -------------- | ---------------------------- | -------------------------------- |
| Sprint (папка) | `001 - Short Description`    | `X - 001 - Short Description`    |
| Task (файл)    | `001 - Short Description.md` | `X - 001 - Short Description.md` |

Префікс `X - ` додавати тільки після явного підтвердження користувача, що sprint/task завершені і правок більше немає.

### Структура папки sprint

- `plan.md` — основний план.
- `tasks/` — task-файли.
- `implementation_docs/` — документація завершених і прийнятих tasks.
- `review.md` — тільки після запиту на фінальне review.

## 4. Implementation Workflow

### Step 1. Receive Scope

- Прочитати `plan.md` поточного sprint і поточний task-файл.
- Перевірити пов'язану документацію в `implementation_docs/`.
- Визначити область змін, винести відкриті питання явно.

### Step 2. Planning

- Підготувати короткий план реалізації, описати підхід, ризики, відкриті питання.
- Не додумувати вимоги і не приймати архітектурні рішення без підтвердження.
- Якщо вимоги неоднозначні — спочатку поставити питання.

### Step 3. Implementation

- Реалізувати тільки погоджений scope.
- Дотримуватись існуючих патернів проєкту.
- Мінімальні точкові зміни.
- Не робити cleanup або refactoring поза межами task.

### Step 4. Verification

- Запустити релевантні перевірки. Якщо щось не запускалося — явно сказати чому.
- Перевірити, що зміни в межах погодженого scope.

### Step 5. Review With User

- Коротко описати зміни, показати результат або де його перевірити.
- Повідомити про запущені перевірки.
- Запитати, чи потрібні правки.

Фінальний review-файл не створювати автоматично.

### Step 6. Task Implementation Documentation

Створювати або оновлювати документацію task **тільки** після явного підтвердження користувача, що task виконана і правок більше немає.

Після такого підтвердження:
- створити файл у `.project_office/sprints/<sprint>/implementation_docs/` у форматі `001 - Task Name.md`;
- описати: що реалізовано, які файли/модулі змінено, важливі рішення та компроміси, які перевірки запущено, що варто знати наступному агенту;
- перейменувати task-файл, додавши префікс `X - `;
- згенерувати commit message через `/generate-commit-message` і показати у чаті.

### Step 7. Final Review (на запит)

Формується тільки після прямого запиту користувача. Зберігати в `.project_office/sprints/<sprint>/review.md`.

Описати: що реалізовано в межах sprint, які tasks завершені, які файли/модулі змінено, які перевірки запущено, які ризики/обмеження залишилися, що врахувати перед наступним sprint.

Формувати на основі task-документації з `implementation_docs/`.

### Step 8. Project Documentation (на запит)

Формувати тільки коли sprint або його погоджена частина завершені, потрібні `implementation_docs/` створені, користувач явно попросив.

Зберігати в `docs/`. Описати: поточну поведінку, реалізований функціонал, важливі рішення та обмеження, основні файли/модулі, що варто знати наступному агенту.

---

## 5. Agent Constraints

Завжди:
- працювати в межах поточного sprint і task;
- берегти існуючі патерни backend/frontend;
- питати перед неоднозначними або архітектурними рішеннями;
- явно повідомляти, які перевірки запущено;
- відділяти факти з файлів від припущень.

Без погодження не робити:
- перехід до наступного sprint або task;
- створення нових sprints, tasks або підзадач;
- перейменування папок sprint або файлів task на `X - `;
- створення або оновлення `implementation_docs/`;
- великі refactoring зміни або зміни поза межами поточної task;
- фінальне review або документацію в `docs/`;
- зміну структури `.project_office/`.
